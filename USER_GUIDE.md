# True 3D Voxel Maze v1.0.0 User Guide

True 3D Voxel Maze は、ブラウザだけで動作する HTML5 版の真の立体迷路アプリです。一般的なダンジョンRPG型の「平面迷路を一人称視点で歩く」アプリではありません。立方体状の solid ボクセル空間の中に empty の通路を掘り、その中を重力に縛られず、上下左右前後の6方向へ向きを変えながら進む迷路です。

This is a browser-based true 3D voxel maze app. It generates a carved tunnel maze inside a solid voxel block and lets the player explore it locally in the browser.

## このアプリでできること

このバージョンでは、次のことができます。

- 立方体状の3Dボクセル迷路をランダム生成する
- 3³、4³、5³、6³、7³、8³ の6種類の論理迷路サイズを選ぶ
- 追加通路の量を0〜30%のスライダーで調整し、一本道感やループ量を調整する
- シードを使って同じ設定の迷路を再生成する
- 生成時に実際に使われたシードを確認し、クリックでコピーする
- 矢印キーまたは画面ボタンで90度単位に向きを変える
- Space で前進、Shift + Space で後退する
- 操作ボタンの色で、その方向に通れるかどうかを確認する
- 右上の立方体ミニマップで現在位置、向き、ゴール位置を確認する
- 一度通った場所をマーカーで確認する
- 壁のベース色とゴール色を変更する
- ゴールに近い壁ほどゴール色に近づくグラデーション表示で、空間の手がかりを得る
- 日本語 / English のUI表示を切り替える

このアプリの中心は、単なる3D風表示ではなく、**迷路データそのものが3次元であること**です。プレイヤーは床と天井の区別に縛られず、現在の正面方向と上方向を持った状態で、6方向のボクセル通路を探索します。

## ローカル処理とプライバシー

このアプリは、単体HTMLとしてブラウザ内で動作します。迷路生成、プレイヤー操作、描画、シード処理、通過済みマーカーの記録は、すべて利用者のブラウザ上で実行されます。

現行版では、迷路データや操作履歴をサーバーに送信する機能はありません。GitHub Pages で公開した場合でも、配信されるのは静的ファイルだけです。アプリ本体は読み込まれた後、利用者のブラウザ内で完結します。

ブラウザのローカルストレージ保存やセーブデータ機能も v1.0.0 では実装していません。ページを再読み込みすると現在の探索状態は失われます。ただし、現在のシードを控えておけば、同じバージョンと同じ生成ロジックの範囲では迷路を再生成できます。

## 対応ブラウザと推奨環境

推奨環境は、デスクトップ版の最新 Google Chrome または Microsoft Edge です。

アプリは標準的な HTML、CSS、JavaScript、Canvas 2D API を使って実装されています。外部ライブラリは使用していません。多くのモダンブラウザで動作する可能性がありますが、開発・確認の主対象は Chromium 系ブラウザです。

スマートフォンやタブレットでも操作できるよう、画面幅が狭い場合は右ペインが迷路の下へ回り込みます。ただし、3D迷路の視認性や操作性を考えると、最初はPCまたは横向きタブレットでの利用を推奨します。

## 迷路の基本コンセプト

このアプリの迷路は、2D迷路を3D表示したものではありません。迷路そのものが3次元の空間として作られています。

基本的な考え方は次の通りです。

- 全体は立方体状のボクセル空間です。
- 最初はすべてのボクセルが solid、つまり固体ブロックです。
- 迷路生成時に通路部分だけを empty、つまり空洞として掘ります。
- プレイヤーは empty ボクセルだけを移動できます。
- 隣接ボクセルが empty なら通路、solid なら壁です。
- ゴールはスタートから遠い位置に設定されます。

プレイヤーには常に「現在位置」「正面方向」「上方向」があります。地球上の通常の迷路のように、画面の上が常に天井で、下が常に床であるとは限りません。上回転・下回転を行うと、以前の床や天井が正面になることもあります。

この性質により、通常の平面迷路とは異なる方向感覚が必要になります。右上のミニマップや通過済みマーカーは、この立体的な迷いやすさを補助するための機能です。

## 基本ワークフロー

通常の使い方は次の通りです。

1. `index.html` をブラウザで開きます。
2. 必要に応じて言語を選びます。
3. 迷路サイズを選びます。
4. 追加通路の量を選びます。
5. 必要に応じてベース色とゴール色を変更します。
6. シードを指定したい場合は、シード欄に入力します。
7. **新しい迷路** を押します。
8. 矢印キーまたは画面ボタンで向きを変えます。
9. Space で前進します。
10. 必要なら Shift + Space で後退します。
11. 右上のミニマップと通過済みマーカーを参考にしながらゴールを目指します。

シード欄が空欄の場合、ランダムシードで迷路が生成されます。その際に使われた完全なシードは、`現在のシード` に表示されます。

## 画面構成

### ヘッダー

画面上部には、アプリ名 `True 3D Voxel Maze` とバージョン `v1.0.0` が表示されます。

### 迷路描画エリア

左側には迷路の一人称視点が表示されます。デスクトップ表示では最大 800 x 600 相当の4:3画面です。16:9ではなく4:3寄りにしている理由は、現在位置から上下左右に通路がある場合でも、開口部が画面外へ逃げすぎないようにするためです。

迷路画面の左上には、現在位置、正面方向、上方向、歩数のHUDが表示されます。右上には立方体ミニマップが表示されます。

### 右ペイン

右ペインは上から順に、次の構成です。

1. 操作ボタン
2. 探索情報
3. 迷路の設定

横幅が狭い場合、右ペインは自動的に迷路描画エリアの下に移動します。

## 操作方法

キーボード操作は次の通りです。

| 操作 | 動作 |
|---|---|
| ↑ | 上方向へ90度回転 |
| ↓ | 下方向へ90度回転 |
| ← | 左方向へ90度回転 |
| → | 右方向へ90度回転 |
| Space | 現在の正面方向へ前進 |
| Shift + Space | 現在の正面方向の逆へ後退 |

画面上の操作ボタンでも同じ操作ができます。スマートフォンやタブレットでは、ボタン操作を使ってください。

このアプリでは、左右回転だけでなく、上下回転も重要です。上下回転により、現在の「上方向」が変化します。たとえば上回転を行うと、これまでの上方向が新しい正面方向になります。

---

## 操作ボタンの色分け

操作ボタンは、現在位置と現在向きを基準に、各方向へ進めるかどうかを色で示します。

| 表示 | 意味 |
|---|---|
| 緑系 | その方向は empty で通行可能 |
| 赤系 | その方向は solid で通行不可 |

ボタンの色は、現在の正面・背面・上・下・左・右を基準に計算されます。世界座標の上方向ではなく、プレイヤーから見た現在の方向です。

この表示は、立体迷路で現在セル周辺の通路構造を把握するための補助です。特に上下方向の通路は一人称視点だけでは分かりにくいことがあるため、ボタン色を確認すると現在位置の接続状態を把握しやすくなります。

## 探索情報

右ペインの「探索情報」には、次の5項目が表示されます。

| 項目 | 意味 |
|---|---|
| 位置 | 現在いるボクセル座標です。 |
| 正面 | 現在向いている方向です。 |
| 上方向 | 現在の画面上方向に対応する3D方向です。 |
| ゴール | ゴールのボクセル座標です。 |
| 歩数 | 前進または後退で移動した回数です。回転は歩数に含みません。 |

座標は、実ボクセル空間上の座標です。論理迷路サイズが `N³` の場合、実ボクセル空間は `2N+1` の大きさになります。たとえば `4³ rooms / 9³ voxels` では、座標は 0〜8 の範囲を取ります。

プレイヤーが通常立つ論理ルームは奇数座標にあり、ルーム間の通路はその中間の偶数座標を通ります。

## 迷路の設定

「迷路の設定」には、迷路生成と表示に関する設定がまとまっています。

設定項目は次の通りです。

- 言語
- サイズ
- 追加通路
- シード
- 現在のシード
- 新しい迷路
- 通過済みマーカー表示
- カラーリング

`新しい迷路` を押すと、サイズ、追加通路、シード設定に基づいて迷路が作り直されます。カラーリングと通過済みマーカー表示は、生成後でも表示に反映されます。

---

## サイズ

v1.0.0 では、次の6サイズを採用しています。

| 表示 | 論理ルーム数 | 実ボクセル空間 | 位置づけ |
|---|---:|---:|---|
| 3³ rooms / 7³ voxels | 27 rooms | 7 x 7 x 7 | 小さめ、確認用 |
| 4³ rooms / 9³ voxels | 64 rooms | 9 x 9 x 9 | 標準サイズ |
| 5³ rooms / 11³ voxels | 125 rooms | 11 x 11 x 11 | 大きめ |
| 6³ rooms / 13³ voxels | 216 rooms | 13 x 13 x 13 | 大きめ |
| 7³ rooms / 15³ voxels | 343 rooms | 15 x 15 x 15 | かなり大きめ |
| 8³ rooms / 17³ voxels | 512 rooms | 17 x 17 x 17 | 現行版の最大サイズ |

論理ルームは迷路生成上のセルです。実際の描画と移動には、その間をつなぐ中間ボクセルも含まれます。そのため、論理サイズ `N³` に対し、実ボクセル空間は `2N+1` になります。

サイズを大きくすると、探索空間が急激に広がります。初めて遊ぶ場合は、4³ または 5³ から試すことを推奨します。7³、8³ は長めの探索や難度確認向けです。

## 追加通路

追加通路は、迷路生成後にどの程度余分な接続を掘るかを指定する設定です。0%から30%まで、1%刻みのスライダーで指定できます。初期値は7%です。

目安は次の通りです。

| 値 | 傾向 |
|---:|---|
| 0% | DFSで作った完全迷路に近い。ループなし。 |
| 5%前後 | 少しだけ迂回路ができる。小さめの迷路では十分効く。 |
| 7% | v1.0.0 の初期値。小〜中サイズ向けの標準的な値。 |
| 15%前後 | 分岐・迂回路がかなり増える。大きめの迷路で一本道感を弱めたい場合に向く。 |
| 20〜30% | ループが多く、開いた迷路になりやすい。迷路というより洞窟状ネットワークに近づく。 |

最初の迷路はDFSで生成されます。DFSだけで作った迷路は、全体として木構造になり、スタートから各地点への経路が1本に決まります。この性質は「迷路らしさ」を作る一方で、探索感としては長い一本道と行き止まりの組み合わせに見えることがあります。

追加通路を増やすと、未接続だった隣接ルーム同士の間が追加で掘られます。これによりループや迂回路が生まれ、分岐の多い迷路になります。

追加通路の設定値は、シードにパーセント値として埋め込まれます。完全なシード文字列を使って再生成した場合、追加通路スライダーもシード内の値に合わせて自動変更されます。

## シードと現在のシード

### シード欄

シード欄に文字列を入れて `新しい迷路` を押すと、その文字列を元に迷路を生成します。シード欄が空欄の場合は、アプリがランダムなベースシードを作成します。

### 現在のシード

迷路生成に実際に使われた完全なシードは、`現在のシード` に表示されます。ここをクリックすると、クリップボードへコピーできます。クリップボードへの直接コピーが使えない環境では、シード文字列がメッセージとして表示されます。

v1.0.0 の完全なシードは、次のような形式です。

```text
T3DM1|extra=7|seed=maze-lxxxx-xxxxx
```

主な要素は次の通りです。

| 要素 | 意味 |
|---|---|
| `T3DM1` | True 3D Maze v1 系のシード形式を示す接頭辞です。 |
| `extra=...` | 追加通路率です。0〜30 の整数が入ります。 |
| `seed=...` | 乱数生成に使うベースシードです。 |

`extra` の値はパーセント表記ではなく、整数だけを保存します。たとえば `extra=7` は追加通路 7% を意味します。

### 互換性について

v1.0.0 は最初の公開版です。シード形式に過去互換性はありません。また、将来のバージョンで迷路生成ロジックを変更した場合、同じシードでも同じ迷路にならない可能性があります。

長期的に完全に同じ迷路を保存したい場合は、将来的に専用の迷路データ保存形式を追加する必要があります。v1.0.0 のシードは、主に同一バージョン内での再現・共有・デバッグ用の機能です。

## 通過済みマーカー

通過済みマーカーは、一度通った empty ボクセルを示す補助表示です。v1.0.0 ではデフォルトでONです。

マーカーは、通過済み空間の中央付近に小さな青系の球として表示されます。ただし、純粋に物理的な3D位置の中心に描くと、直線通路ではすべてのマーカーが消失点上に重なりやすくなります。そのため、v1.0.0 では視認性を優先し、マーカーだけ少し下側にずらして表示しています。

また、下方向に近い壁面がある場合は、マーカーの下に薄い影を描きます。これは本格的な照明計算ではなく、通路内で点列の位置を読み取りやすくするための簡易表現です。

マーカー表示は `通過済み / マーカーを表示` のチェックでON/OFFできます。純粋に迷路の難しさを楽しみたい場合はOFFにできます。

## カラーリング

カラーリングでは、迷路壁面の色を変更できます。

| 項目 | 意味 |
|---|---|
| ベース | ゴールから遠い場所の基準色です。 |
| ゴール | ゴールに近い場所の基準色です。 |

描画時には、壁面上の点からゴールまでの距離を使い、ベース色とゴール色をブレンドします。ゴールに近い壁ほどゴール色に近くなり、遠い壁ほどベース色に近くなります。

さらに、壁面の向きに応じて明度を少し変えています。正面に近い壁はやや明るく、横方向の壁はやや暗くなります。これにより、単色の壁だけでは分かりにくい立体感を補助しています。

この色変化は、厳密な照明シミュレーションではありません。迷路内で方向感覚とゴール方向の手がかりを得るための表示上の工夫です。

## ゴール表示とミニマップ

### 迷路内のゴール表示

ゴールが視界内にあり、壁に遮られていない場合、迷路内にゴールマーカーが表示されます。ゴール色はカラーリングの `ゴール` 設定と連動します。

ゴールが壁の向こうにある場合は、迷路内には表示されません。

### 右上の立方体ミニマップ

右上のミニマップは、迷路全体を小さなワイヤーフレーム立方体として表示します。

表示内容は次の通りです。

- 立方体全体：迷路空間全体
- 青い点：現在位置
- 青い短い矢印：現在向いている方向
- ゴール色の点：ゴール位置
- 距離表示：現在位置からゴールまでの直線距離の概算

矢印の長さはゴールまでの距離には連動しません。常に見える長さで描かれ、現在向いている方向だけを示します。ゴールまでの距離は、別途テキストで表示します。

## 迷路生成ロジックの概要

v1.0.0 の迷路生成は、大きく次の流れです。

1. 指定された論理サイズ `N³` のルーム空間を用意する
2. 実ボクセル空間 `2N+1` をすべて solid として初期化する
3. DFS / recursive backtracker 方式で論理ルームを訪問する
4. 訪問時に、現在ルーム、次ルーム、両者の中間ボクセルを empty にする
5. 追加通路設定に応じて、未接続の隣接ルーム間を追加で掘る
6. スタートから到達できる empty 空間を探索し、遠い論理ルームをゴールにする

DFSだけで生成すると、迷路グラフは木構造になります。木構造では任意の2点間の経路が1本だけです。そのため、迷路としては正統的ですが、3D空間では「長く続く一本道」に感じられることがあります。

追加通路は、この一本道感を弱めるための後処理です。

---

## solid--empty ボクセルモデル

このアプリでは、壁を薄い面として持つのではなく、ボクセルそのものを solid または empty として管理しています。

初期状態はすべて solid です。迷路生成時に通路だけを empty にします。描画時には、プレイヤーの視線方向へレイを飛ばし、最初にぶつかった solid ボクセル面を壁として描きます。

この方式の利点は、次の通りです。

- 空洞の中を進んでいる感覚が出やすい
- 壁と通路の区別が明確になる
- 3D方向の分岐をボクセル構造として扱いやすい
- 移動判定が「隣接ボクセルが empty かどうか」で済む

一方で、ボクセル境界の輪郭線を描く際には、どの線が本当に見える構造線なのかを慎重に扱う必要があります。v1.0.0 では、この点に対して簡易的な可視判定を追加しています。

---

## 追加通路の技術的な意味

追加通路は、DFS生成後に、隣り合っているがまだつながっていない論理ルーム間の中間ボクセルを、一定確率で empty にする処理です。

たとえば、論理ルーム A と B が隣接していても、DFSで直接接続されなかった場合、その間の中間ボクセルは solid のままです。追加通路処理では、この中間ボクセルを確率的に掘ります。

追加通路率が高いほど、次の傾向が強くなります。

- ループが増える
- 迂回路が増える
- 行き止まりの重要度が下がる
- 一方通行的な探索感が弱まる
- ゴールまでの経路候補が増える

ただし、追加通路を増やしすぎると、迷路というより洞窟状のネットワークに近づきます。また、同じ追加通路率でも、迷路サイズが大きくなるほど体感上の効果は弱くなる場合があります。v1.0.0 では、0〜30%の範囲でユーザーが直接調整できるようにしています。

## 描画ロジックの概要

描画は Canvas 2D を使った簡易レイキャストです。Three.js や WebGL は使っていません。

各フレームでは、次のような処理を行います。

1. 現在位置、正面方向、上方向からカメラを作る
2. 各ピクセル方向にレイを飛ばす
3. DDA方式でボクセル空間を進み、最初に当たった solid を探す
4. 当たった壁面の色を計算する
5. 低解像度バッファに壁面を描く
6. バッファを画面サイズへ拡大する
7. 輪郭線を重ねる
8. 通過済みマーカーを描く
9. ゴールマーカーを描く
10. ミニマップとHUDを描く

低解像度バッファを使うのは、Canvas 2D 上で全ピクセルを直接レイキャストすると重くなりやすいためです。輪郭線やマーカーは画面側に重ねることで、線の太さやマーカーの視認性を保っています。

## 輪郭線描画と可視判定

ボクセル壁面だけを塗ると、通路の角や開口部の位置が分かりにくくなることがあります。そのため、v1.0.0 では輪郭線を常時表示します。

輪郭線は、empty と solid の境界面から構造エッジを抽出して描画しています。ただし、単純にエッジの中点だけで可視判定すると、奥にある開口部の線が手前の壁面まで延長されたように見える場合がありました。

v1.0.0 では、この問題を避けるため、線分上の2点、具体的には 1/3 点と 2/3 点の両方が見えている場合だけ、その輪郭線を描くようにしています。これにより、手前の壁に一部隠れている奥の線分を丸ごと描いてしまう問題を抑えています。

この処理は完全な3Dクリッピングではありませんが、現行のボクセル迷路と90度単位の視点操作には十分実用的な軽量処理です。

## 現在の制限事項

v1.0.0 には、次の制限があります。

- セーブデータ機能はありません。
- 途中状態の保存・復元はできません。
- 迷路データのインポート / エクスポート機能はありません。
- シード互換性は将来バージョンで保証されません。
- 迷路生成アルゴリズムはDFS + 追加通路に固定されています。
- 通過済みマーカーは視認性優先の補助表示であり、厳密な物理表現ではありません。
- 影表現は簡易的な疑似表現です。
- 描画はCanvas 2Dベースであり、本格的なWebGL 3Dエンジンではありません。
- 画面酔い対策、感度調整、アニメーション速度調整は未実装です。
- モバイル端末では画面サイズや処理性能によって操作性が変わります。

これらは、最初のリリースで機能を絞るための制限です。アプリの中核であるボクセル迷路生成、一人称探索、ミニマップ、通過済みマーカー、シード再現性は v1.0.0 で実装済みです。

## GitHub Pages で公開する場合

GitHub Pages で公開する場合は、リポジトリ直下に次のファイルを置いてください。

- `index.html`
- `README.md`
- `USER_GUIDE.md`
- `CHANGELOG.md`
- `RELEASE_REVIEW.md`
- `LICENSE`

ビルド手順は不要です。GitHub Pages が `index.html` を静的ファイルとして配信すれば、そのまま動作します。

ローカルで確認する場合も、通常は `index.html` をブラウザで直接開くだけで動作します。

## 今後の拡張候補

今後の拡張候補としては、次のようなものが考えられます。

- 探索状態の保存・復元
- 迷路データそのもののエクスポート / インポート
- 難易度プリセット
- ゴールまでの最短距離表示のON/OFF
- ミニマップの表示モード切替
- マーカー色のカスタマイズ
- アニメーション速度設定
- 画面酔い対策としてのアニメーションOFF設定
- WebGL版またはThree.js版の実験
- スコア、到達時間、最短歩数との比較
- スマートフォン向け操作UIの追加調整

v1.0.0 は、まず「単体HTMLで、真の3Dボクセル迷路として遊べる」ことを優先した初期リリースです。
